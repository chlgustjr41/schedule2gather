# schedule2gather v1.1 Follow-up — Design Spec

**Date:** 2026-07-18
**Status:** Approved by user (brainstorm with visual companion; layout validated via mockup)
**Lands on:** the open `redesign` branch / PR #1 (ships with the redesign as one unit)
**Parent spec:** `2026-07-18-redesign-design.md`

## 1. Scope

Five changes driven by user feedback on the shipped redesign:

1. **Range date selection** in the create flow (toggle between per-day picking and start→end range).
2. **Timeslot screen restructure** — the Me/Group toggle and desktop dual-grid confused the user. Replaced by stacked, clearly-labeled sections with the **group heatmap first**, transposed to horizontal per-date strips ("Layout B, group first" — user-selected mockup).
3. **Who's-free tooltip on the user's own paint grid**, directly above the hovered cell.
4. **Host can finish the vote**: pick a final time, lock painting (server-enforced), show a finalized banner with add-to-calendar; host can reopen.
5. **Smarter garbage collection**: delete events whose dates have all passed and that nobody has visited in 30 days (in addition to the existing 90-day hard cap).

Out of scope: any other layout change, cross-device identity, presence changes.

## 2. UX

### 2.1 Create flow — range toggle

A `SegmentedControl` above the calendar: **Pick days | Date range** (default: Pick days).

- Range mode uses `DayPicker mode="range"`. When a complete from→to range exists, all days in the interval with past days excluded are **merged** into `selectedDates`, then the range draft resets so another range can be added.
- Mode switching never discards existing selections. "Clear all" clears selections and any range draft. Quick-select month rows unchanged and available in both modes.
- The "N selected" count reflects the merged set.

### 2.2 Event page layout (all viewports — dual-grid and Me/Group toggle are removed)

Order: header → share card → ✨ Best times (or Finalized banner, §2.4) → **👥 Group overlap** → **✏️ My times** → comments.

**Group overlap card** (green left-accent `4px solid success`, `Card` base):
- Header "👥 Group overlap", subline "Read-only — darker green = more people free".
- One row per event date: date label ("Mon Jul 20", viewer-TZ) + a horizontal strip of `slotsPerDay` cells (`flex`, equal width, min-width 4px, height ~18px), colored via existing `heatColor(count, total)`.
- Hour marks: time-range start label under the strip's left end, end label under the right end (viewer TZ).
- Legend row: `0 [swatch×5] all free`.
- Tooltip on hover (desktop) or tap (touch): "7:00–7:15 PM · Jacob, Sam · 2/3" above the cell; tap elsewhere dismisses.

**My times card** (orange left-accent `4px solid primary`):
- Header "✏️ My times", subline "Tap or drag to paint when you're free".
- Contains the existing toolbar (week/month, mark-all, clear, undo/redo, paint-mode toggle) and the paint grid. All paint/undo/keyboard/ARIA/pagination behavior is unchanged.
- **Own-cell tooltip:** hovering a cell (while not dragging) shows the existing `CellTooltip` directly above it, listing the names of participants available in that slot (and nothing when none). Suppressed during paint drags, unchanged from the old group-cell behavior — the tooltip moves home, it isn't rebuilt.

`AvailabilityGrid` consequently loses: `layer` state, the SegmentedControl view toggle, `dualGrid`/`useMinWidth`, and the group render path (`renderTable('group', …)` and per-cell group aria/tooltip branches). It renders only the interactive mine-layer table. `useMinWidth.ts` is deleted if nothing else imports it.

### 2.3 Finish vote (host)

- In the Best times card, hosts see a **🏁 Finish vote** primary button (hidden for non-hosts; hidden when already finalized).
- Tapping opens `FinalizeSheet` (BottomSheet): the current top-3 `bestWindows` as radio rows (same labels as the panel), plus a **"Custom time…"** row that expands a date select (the event's dates) and start/end time selects bounded to the event's time range and slot grid — honoring "any window selectable". A **Finish vote** confirm button; when no best windows exist the suggestions are absent but Custom still works.
- Confirm writes `finalized` (§3.1). All viewers live-update via the existing event subscription.

### 2.4 Finalized state (all viewers)

- The Best times panel is replaced by a **FinalizedBanner** card (success-tinted): "✅ Finalized — Wed Jul 22, 7:00–8:30 PM" (viewer TZ) + **Add to calendar** button opening the existing `ExportSheet` for the final window + host-only ghost "Reopen voting" button.
- The My times card becomes read-only: toolbar hidden, cells non-interactive (no pointer/keyboard handlers, `cursor-default`), subline replaced by "Voting closed — the time is locked in."
- Group heatmap and comments remain live. Reopen restores everything.
- **New visitors to a finalized event skip the join screen entirely** — the finalized page (banner, heatmap, comments) renders without joining, since painting is closed and rules would reject the participant write anyway. Commenting still requires having joined; the comment input shows "Voting is closed" for non-participants.

### 2.5 Failure surfaces

- Finalize/reopen write failures render an inline error line inside the sheet/banner ("Couldn't save — try again"); no crash.
- A paint attempt racing a finalize is rejected by security rules; the optimistic local state is corrected by the next participants snapshot. No modal.
- `lastVisitedAt` writes are fire-and-forget and silent on failure.

## 3. Data, Rules, Backend

### 3.1 Event document additions (no migration; fields optional)

```
events/{slug}
  ├── finalized?: { startSlot: number, endSlot: number, finalizedAt: Timestamp } | null
  └── lastVisitedAt?: Timestamp
```

`EventDoc` in `eventService.ts` extends accordingly. New service functions:
- `finalizeEvent(slug, { startSlot, endSlot })` — setDoc merge with `finalizedAt: serverTimestamp()`.
- `reopenEvent(slug)` — sets `finalized: null`.
- `touchLastVisited(slug, current?: Timestamp)` — writes `lastVisitedAt: serverTimestamp()` only when the current value is absent or older than 24 h (client-side throttle); called from `EventPage` once the event loads.

### 3.2 Firestore rules

- `events/{id}` update allowed when:
  - auth'd, `affectedKeys().hasOnly(['lastVisitedAt'])` — anyone (incl. anonymous) may touch the visit timestamp; **or**
  - `request.auth.uid == resource.data.ownerUid` and `affectedKeys().hasOnly(['finalized'])` — host finalize/reopen; **or**
  - the existing `ownerEmail` carve-out (unchanged).
- `events/{id}/participants/{pid}` writes additionally require
  `get(/databases/$(database)/documents/events/$(id)).data.get('finalized', null) == null` —
  painting is blocked server-side while finalized. (One extra read per availability write; acceptable at this scale.)

### 3.3 Garbage collection

`functions/src/lib/gc.ts` — pure `shouldDeleteEvent(event, now): boolean`:

- `true` if `expiresAt <= now` (existing 90-day hard cap), **or**
- event has calendar dates (`mode !== 'weekdays_recurring'`), every date's day has fully passed (last date < today in UTC), **and** `(lastVisitedAt ?? createdAt)` is more than **30 days** before `now`.
- `weekdays_recurring` events: hard cap only.

`cleanupExpiredEvents` iterates events (collection scan as today), applies the predicate, and uses the existing `recursiveDelete`. Firestore index needs: none (scan unchanged).

## 5. Testing

- **Unit (new):** `dateRange` (merge, past-day exclusion, overlap dedupe); `gc.shouldDeleteEvent` (hard cap, all-dates-passed + idle, fresh visit blocks deletion, `createdAt` fallback, recurring-mode cap-only) in `functions/src/__tests__`.
- **Existing suites** stay green; grid tests unaffected (logic unchanged, only rendering paths removed).
- **E2E additions:** group strips render one row per date; own-cell hover tooltip shows names; host finalize → banner + locked grid + rules-level write rejection; reopen restores painting; range toggle selects an interval.

Same hierarchical multi-agent pattern as the redesign: parallel wave for the independent pure parts (`dateRange` lib, `gc` predicate, rules+service layer), sequential wave for the UI restructure (GroupHeatmap → AvailabilityGrid shrink → EventPage assembly → finalize UI), review gate against this spec, docs, then full gate + browser E2E before pushing to the open PR.
